# Phần 6.2b — Port OAuth/OIDC sang auth-service

> Tách nhỏ khỏi 6.2 để mỗi commit gọn. Ở đây ta chuyển OAuth (Phần 3) vào auth-service,
> thích ứng theo **model credential** + **database-per-service**.

---

## 6.2b.1 — Vì sao OAuth phải sửa, không chỉ copy?

Ở monolith, OAuth tạo thẳng một `User` (có cả profile lẫn role) trong một DB. Giờ:

- **auth-service** không có bảng `users` (profile) — nó chỉ có `credentials`.
- User mới đăng nhập OAuth ⇒ phải **tạo profile ở user-service** (gọi mạng) rồi tạo `credential`
  trỏ về `userId` đó — **y hệt luồng register** ở 6.2 (kể cả bù trừ khi lỗi).

Vậy OAuth nay cũng là một **luồng liên service**, không còn "một phát ghi DB" như trước.

## 6.2b.2 — Hai thay đổi schema (app_auth)

```prisma
model Credential {
  ...
  passwordHash String?        // NULLABLE: user OAuth không có mật khẩu
  oauthAccounts OAuthAccount[]
}

model OAuthAccount {           // liên kết (provider, sub) ↔ credential
  provider          String
  providerAccountId String
  userId            String
  @@unique([provider, providerAccountId])
}
```

`passwordHash` thành nullable ⇒ `login` phải chặn: tài khoản **chỉ-OAuth** (hash null) không được
đăng nhập bằng mật khẩu (xem `auth.service.login`). Chạy `pnpm migrate:auth` để tạo migration này.

## 6.2b.3 — Account linking (giữ nguyên tinh thần Phần 3)

`oauth.service.loginWithOAuth` xử lý 3 tình huống:

1. **Đã liên kết** `(provider, sub)` → chính là login → cấp token.
2. **Chưa liên kết nhưng email đã tồn tại** → *auto-link* — nhưng **chỉ khi** provider báo
   `email_verified` (nếu không, kẻ gian tạo tài khoản provider với email của bạn để chiếm tài khoản).
3. **Hoàn toàn mới** → tạo profile (user-service) + credential (không mật khẩu) + link.

Sau đó **tái dùng `token.service`**: cấp access + refresh token *của hệ thống ta*. OAuth chỉ để
**xác minh danh tính**; sau đó user dùng token nội bộ như mọi user khác.

## 6.2b.4 — Đường đi qua gateway

`redirect_uri` trỏ về **gateway** `http://localhost:4000/api/auth/oauth/:provider/callback`.
Browser → gateway (:4000) → proxy → auth-service (:4001). auth-service `res.redirect(...)` sang
provider; provider gọi lại gateway → proxy về auth-service. `http-proxy-middleware` chuyển tiếp cả
302 redirect lẫn cookie, nên FE (Phần 4) **không đổi gì**.

## 6.2b.5 — Thử

```bash
docker compose up -d keycloak          # realm "app" tự import
# điền KEYCLOAK_* vào apps/auth-service/.env  (đã có sẵn trong .env.example)
# mở trình duyệt:
open http://localhost:4000/api/auth/oauth/keycloak/login
# login user test: tester / password123
```

> Nút Google/Keycloak trên FE giờ hoạt động trở lại. Google thì điền `GOOGLE_CLIENT_ID/SECRET`.

# Phần 6.5 — Service Discovery & Cấu hình tập trung

> URL service đang bị hardcode/rải rác (`http://localhost:4002` ở nhiều nơi, mỗi service một `.env`).
> Commit này gom việc "service X ở đâu?" về **một registry** — mầm mống của service discovery.

---

## 6.5.1 — Vấn đề: địa chỉ cứng & config rải rác

- auth-service dựng URL user-service từ `env.USER_SERVICE_URL`; gateway lại có `AUTH_SERVICE_URL`,
  `USER_SERVICE_URL` riêng; gRPC thì `USER_SERVICE_GRPC`… Mỗi nơi một kiểu.
- Đổi port/host user-service ⇒ sửa **nhiều file**. Khi số service tăng, đây là ác mộng.
- Ở production, địa chỉ **động** (container scale lên/xuống, IP đổi liên tục) — không thể hardcode.

## 6.5.2 — Registry: hỏi "ở đâu" thay vì tự dựng URL

`packages/shared/src/registry.ts` là **danh bạ** duy nhất:

```ts
serviceRegistry.userHttp();  // "http://localhost:4002"
serviceRegistry.userGrpc();  // "localhost:4092"
serviceRegistry.authHttp();  // "http://localhost:4001"
```

Mọi caller giờ hỏi registry:

- `user-client` (REST) và `user-grpc-client` → `serviceRegistry.userHttp()/userGrpc()`.
- gateway → `serviceRegistry.authHttp()/userHttp()`.

Địa chỉ mặc định gom **một chỗ** (`DEFAULTS`) — hết magic number rải rác. Đây là **cấu hình tập trung**
cho "topology" của hệ thống.

## 6.5.3 — Vì sao đây là "mầm" service discovery thật

Ruột registry hiện đọc từ ENV + default. Điểm mấu chốt: **chữ ký hàm không đổi**. Ở production bạn
chỉ THAY phần thân:

```ts
// dev (hiện tại)
userHttp: () => process.env.USER_SERVICE_URL ?? "http://localhost:4002",

// production với Consul / k8s — code GỌI y nguyên:
userHttp: () => consul.resolve("user-service"),
// hoặc dựa vào DNS nội bộ k8s: "http://user-service.default.svc.cluster.local"
```

`auth-service`, `gateway`… **không phải sửa một dòng** khi bạn nâng cấp cách khám phá service. Đó
chính là giá trị của việc *tách điểm phân giải địa chỉ* ra khỏi code nghiệp vụ.

## 6.5.4 — Bảng đối chiếu các cách discovery (để bạn biết đường đi tiếp)

| Cách | Khi nào dùng | Ghi chú |
| --- | --- | --- |
| **Static/env** (đang dùng) | dev, hệ nhỏ | đơn giản; địa chỉ tĩnh |
| **DNS nội bộ (k8s Service)** | chạy trên Kubernetes | k8s tự phân giải tên → IP pod, load-balance sẵn |
| **Consul / etcd** | ngoài k8s, cần health-check + KV config | service tự đăng ký; client query |
| **Service mesh (Istio/Linkerd)** | quy mô lớn | discovery + mTLS + retry ở sidecar |

## 6.5.5 — Còn config (không phải địa chỉ) thì sao?

Ngoài "ở đâu", còn "tham số gì" (feature flag, timeout, secret). Hướng đi: **config server** tập
trung (Consul KV, AWS AppConfig, Spring Cloud Config…) thay cho nhiều `.env`. Ở tutorial ta dừng ở
registry cho địa chỉ; secret vẫn ở `.env` mỗi service (Phần 9 bàn quản lý secret ở production).

> Tiếp theo **6.6**: giờ request đi xuyên gateway → auth → user (REST+gRPC), ta cần **nhìn thấy**
> cả hành trình đó bằng distributed tracing (OpenTelemetry + Jaeger).
